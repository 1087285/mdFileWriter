## 2. 不具合解析結果

### 不具合 #2: WYSIWYG表示でのtrailing-space改行非認識

- **根本原因工程**: 03 - 詳細設計
- **根本原因の説明**:
  詳細設計書（v1.2.0）の `preprocessMarkdown()` 仕様（§4.3.3a）において、trailing-space（行末スペース2個）の解決策として「`/ {2,}\n/g` → `'  \n'` で入力を正規化して渡す」方式が定められた。しかしこの方式は、**Toast UI Editor の `setMarkdown()` に渡す文字列中のスペースを保護することしか保証しない**。

  Toast UI Editor の WYSIWYG エンジン（ProseMirror ベース）が、`setMarkdown()` で受け取ったMarkdown文字列を内部でHTML DOMに変換する際、行末スペースを `<br>` としてレンダリングするかどうかは別問題である。実際、多くのWYSIWYGエディタはMarkdown→DOM変換時にProseMirror内部でtrailing-spaceを除去または無視するため、`preprocessMarkdown` で文字列を保護しても WYSIWYG 上の表示には反映されない。

  詳細設計書自身が「実装工程で検証・確定すること」と記述しており（§4.3.3a）、本アプローチが WYSIWYG レンダリングにまで効果を持つかどうかを設計段階で検証・明確化できていなかった。このため、不十分な解決策が確定仕様として設計書に記載されたことが根本原因と判断する。

- **見逃し工程**:
  - **04 - 実装**: `preprocessMarkdown` を実装したが、実際の WYSIWYG 表示での `<br>` レンダリングを手動確認せずにリリース候補とした。詳細設計の "実装工程で検証・確定すること" という指示を遂行しなかった。
  - **05 - 単体テスト**: テストケース IT-MD-001〜004 は「`setMarkdown` に渡される文字列がスペース2個を維持しているか」のみ検証しており、WYSIWYG 上での `<br>` レンダリングは JSDOM の制約により検証不能として除外された。この観点の欠落が見逃しとなった。
  - **06 - 結合テスト**: 同上（IT-MD-001〜004 は文字列の正規化のみを確認）。WYSIWYG 実レンダリングはシステムテストに委ね、テスト設計から除外された。
  - **07 - システムテスト**: ST-E03 の Manual 確認（Pending）が未実施のまま、WYSIWYG 本体の表示動作が検証されなかった。

- **見逃しの理由（共通）**:
  JSDOM 環境では Toast UI Editor の WYSIWYG レンダリングを実動作確認できないため、各テスト工程が「文字列の前処理ロジック検証で代替」という方針をとった。結果として「実際に WYSIWYG 上で改行が見えるか」という最終確認が Windows 実機（Manual）にのみ委ねられ、その Manual テストが未実施のまま不具合として残った。

- **発見工程**: 開発中の手動確認またはユーザー報告（バグ #2 として `bug/bugs.md` に記録）

- **証拠**:

- **再発防止策**:

  | 工程 | 対策 |
  |------|------|
  | **03 詳細設計** | サードパーティライブラリのレンダリング動作を前提とする仕様を定める際は、「XXXライブラリがYYY仕様を満たすこと」を明示的に検証項目として設計書に記載する。"実装工程で確認" の委任を行う場合は、確認方法・合否基準・確認期限を合わせて明示すること |
  | **04 実装** | 詳細設計から "実装工程で検証・確定すること" が付いた項目は、実装後に必ず動作確認（手動またはE2E）を行い、その結果を実装記録に記述すること。確認できない場合は担当者にエスカレーションし、未確認のままリリース候補にしないこと |
  | **05 単体テスト・06 結合テスト** | JSDOM 制約で確認不能なレンダリング観点については、「未確認」として明示的にトラッキングし、代替確認手段（Playwright・Spectron・スクリーンショット差分等）の適用検討を記録すること。テスト除外理由の記録を必須とすること |
  | **07 システムテスト** | Manual:Pending のテスト項目は、リリース前に完了させることをリリース判定の条件に含めること。Pending が残る場合は「既知のリスク」として記録し、承認者の sign-off を得ること |

  **技術的修正案（Bug #2 解消のための実装変更案）**:

  以下のいずれかの対策を `04 実装` 工程で検討・適用すること。

  1. **`preprocessMarkdown` での変換強化**: `setMarkdown()` 呼び出し前に trailing-space を `<br>` に変換する。
     ```javascript
     function preprocessMarkdown(content) {
         // trailing-space（2スペース以上 + 改行）を <br> タグに変換
         return content.replace(/ {2,}(\n)/g, '<br>$1');
     }
     ```
     > ⚠️ `setMarkdown()` が HTML タグを受け付けるかは要検証。WYSIWYG モードでは受け付ける可能性あり。

  2. **Toast UI Editor の customHTMLRenderer / plugins 利用**: `markdown-it` のオプション `breaks: true`（softbreak を `<br>` にする）または trailing-space 対応プラグインを活用する。
     ```javascript
     editorInstance = new Editor({
         ...
         customHTMLRenderer: {
             softbreak: () => ({ type: 'html', content: '<br>' })
         }
     });
     ```

  3. **Markdown→HTML 変換を外部ライブラリで事前実施**: `marked` / `remark` 等の CommonMark 準拠パーサーで Markdown を HTML に変換し、`editor.setHTML()` に渡す。

## 4. プロジェクト全体的な品質課題の考察

本プロジェクトにおいて観察された品質課題は以下のとおりである。

1. **サードパーティライブラリへの動作依存の検証不足**  
   Toast UI Editor（ProseMirror ベース）は WYSIWYG モードでの CommonMark の trailing-space 対応について公式ドキュメントに明記されていない可能性が高い。ライブラリ採用時に "CommonMark 準拠" の範囲を明確に調査せず、前処理によるワークアラウンドを設計した結果、根本的な解決に至らなかった。

2. **"後続工程で確認" の連鎖的未実施**  
   詳細設計の "実装工程で検証・確定すること" → 実装での未確認 → 単体テストの JSDOM 制約による除外 → 結合テストの同除外 → システムテストの Pending という連鎖が発生した。各工程で "次の工程に委ねる" が積み重なると、最終確認が永遠に未実施となるリスクがある。各工程で "自工程内で確認できない場合の代替手段または受け渡し条件" を明示するプロセスが必要。

3. **WYSIWYG実動作のテスト環境整備不足**  
   開発環境が Linux Dev Container に固定されており、Windows 実機での GUI テストが困難な状況がある。Playwright Electron を活用したクロスプラットフォームE2Eテスト環境の整備、または CI での Windows ビルド・テスト環境の構築を検討することで、JSDOM で確認できない観点をカバーできる。
