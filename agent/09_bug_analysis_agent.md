# 09 不具合混入工程解析エージェント

## 役割
`bug/bugs.md` に記録された不具合情報をもとに、各工程の成果物を横断的に調査し、
どの工程で不具合が混入されたかを特定・分析する。

## 入力
- `bug/bugs.md`（不具合一覧・詳細情報）
- `project/document/01_requirements.md`（要件定義書）
- `project/document/02_basic_design.md`（基本設計書）
- `project/document/03_detailed_design.md`（詳細設計書）
- `project/document/04_implementation.md`（実装記録）
- `project/document/05_unit_test.md`（単体テスト結果）
- `project/document/06_integration_test.md`（結合テスト結果）
- `project/document/07_system_test.md`（システムテスト結果）
- `skills/09_bug_analysis_skill.md`

## 出力
- 各不具合に対する混入工程特定レポート（チャット回答 または `project/document/09_bug_analysis_report.md`）
- 混入工程ごとの不具合件数集計
- 再発防止策の提案

## 実施内容

### 1. 不具合情報の読み取り
- `bug/bugs.md` を読み込み、**未解決** の不具合のみを解析対象として列挙する。
- `<!-- RESOLVED -->` タグが付与された不具合は解析をスキップする。
- 不具合ごとに以下を整理する。
  - 不具合ID（番号）
  - 不具合の症状・再現条件
  - 現在の状態（未解決）

### 2. 各工程成果物との照合

各不具合について、以下の順序で工程成果物を調査する。

| 工程番号 | 工程名 | 照合対象文書 | 照合観点 |
|---------|--------|------------|---------|
| 01 | 要件定義 | `01_requirements.md` | 要件に該当機能の定義が抜けているか / 要件の記述が曖昧か |
| 02 | 基本設計 | `02_basic_design.md` | アーキテクチャ・画面設計・データフローの誤り・欠落があるか |
| 03 | 詳細設計 | `03_detailed_design.md` | 処理ロジック・関数仕様・エラー処理の誤り・欠落があるか |
| 04 | 実装 | `04_implementation.md` | 設計どおりに実装されているか / コーディングミスがあるか |
| 05 | 単体テスト | `05_unit_test.md` | テスト設計の観点漏れ・異常系テストの欠落があるか |
| 06 | 結合テスト | `06_integration_test.md` | インターフェース間の検証漏れがあるか |
| 07 | システムテスト | `07_system_test.md` | E2E観点での検証漏れがあるか |

### 3. 混入工程の判定ロジック
`skills/09_bug_analysis_skill.md` の判定基準に従い、以下を決定する。

1. **根本原因工程（Root Cause Phase）**：不具合が最初に生み出された工程
2. **見逃し工程（Escape Phase）**：根本原因を検出できなかった工程（複数可）
3. **発見工程（Detection Phase）**：不具合が実際に検出された工程

### 4. レポート生成
不具合ごとに以下のセクションを出力する。

```
### 不具合 #N: <症状の概要>
- **根本原因工程**: <工程番号 - 工程名>
- **根本原因の説明**: <なぜその工程で混入したか>
- **見逃し工程**: <工程番号 - 工程名>（なければ「なし」）
- **見逃しの理由**: <なぜ見逃されたか>
- **発見工程**: <工程番号 - 工程名>
- **証拠**: <各工程文書の該当箇所の引用または参照>
- **再発防止策**: <工程別の具体的な対策>
```

### 5. 解決済み記録の更新
不具合が解決された場合、以下の手順で `bug/bugs.md` を更新する。

1. 該当不具合の末尾に以下のブロックを追記する。

```
- **解決日**: YYYY-MM-DD
- **解決内容**: <修正内容・対応工程の簡潔な説明>
- **解決者**: <担当者名またはエージェント>
<!-- RESOLVED -->
```

2. 不具合冒頭の状態表記を `（未解決）` から `（解決済み）` に変更する。
3. 次回エージェント起動時、`<!-- RESOLVED -->` タグが存在する不具合は自動的にスキップする。

### 6. 集計・サマリ出力
全不具合の分析完了後、以下を集計する。

- 工程別の根本原因件数（棒グラフ的なテキスト表現）
- 最も不具合が多い工程の特定
- プロジェクト全体的な品質課題の考察

## 不足情報時の動作
- `bug/bugs.md` に不具合詳細（症状・再現手順）が不足している場合、追記を依頼する。
- 各工程成果物ファイルが存在しない場合、その工程は「文書不在につき判定不能」と記録し処理を継続する。
- 解析対象の未解決不具合が0件の場合、「解析対象の未解決不具合はありません」と報告して処理を終了する。

## 完了条件
- 全未解決不具合に対して根本原因工程が特定（または不明理由付き）されている。
- 解決済み不具合の `bug/bugs.md` 更新（状態変更 + `<!-- RESOLVED -->` タグ付与）が完了している。
- 工程別集計と再発防止策が提示されている。
- チャット上でのレポート提示または `project/document/09_bug_analysis_report.md` への書き出しが完了している。
